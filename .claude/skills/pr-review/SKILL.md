# PR Review Checklist — Rust Web App

Проведи полное ревью всех изменённых файлов. Используй Bash для `git diff main` и прочти каждый изменённый файл.

## 1. Domain Layer (критично)

- [ ] **Нет инфраструктурных зависимостей** — domain/ не импортирует sqlx, lapin, redis, axum
- [ ] **Value Objects валидируют себя при создании** — конструктор возвращает `Result<Self, DomainError>`
- [ ] **Invariants Aggregate Root** — все мутации идут через методы агрегата, не через прямое изменение полей
- [ ] **Domain Events** — если состояние изменилось, событие поднято через `aggregate.domain_events()`
- [ ] **Нет паник в домене** — `unwrap()`, `expect()`, `panic!()` запрещены
- [ ] **Ошибки типизированы** через `thiserror`, не `String` / `anyhow`

## 2. Application Layer

- [ ] **Command ≠ Query** — handler либо меняет состояние, либо читает, никогда оба
- [ ] **Idempotency** — команды учитывают повторную обработку (особенно из очереди)
- [ ] **Transaction boundary** — вся бизнес-операция в одной транзакции (включая запись domain events в outbox)
- [ ] **Нет бизнес-логики в handler'ах** — только оркестрация, вся логика в домене

## 3. Infrastructure

- [ ] **Все SQL через sqlx макросы** — `sqlx::query!` / `sqlx::query_as!`, не raw строки
- [ ] **Миграции добавлены** — если изменилась схема БД
- [ ] **Repository реализует trait из домена** — не наоборот
- [ ] **Connection pooling** — используется пул, не отдельное соединение на запрос
- [ ] **Retry на внешние вызовы** — RabbitMQ publish, Redis операции имеют retry logic

## 4. API Layer

- [ ] **DTO отдельно от Domain** — нет прямого возврата доменных сущностей в HTTP response
- [ ] **Валидация входных данных** — `validator` crate на всех входящих DTO
- [ ] **HTTP статусы корректны** — 201 для создания, 404 для not found, 400 для валидации, 409 для конфликтов
- [ ] **Нет чувствительных данных в ответах** — пароли, токены, внутренние ID не утекают
- [ ] **Middleware** — auth проверяется до handler'а, не внутри

## 5. Безопасность (блокер для мержа)

- [ ] **Нет секретов в коде** — никаких хардкод паролей, ключей, токенов
- [ ] **SQL injection невозможен** — параметризованные запросы через sqlx
- [ ] **Нет логирования PII** — email, телефон, пароль, токен не попадают в логи
- [ ] **Авторизация проверена** — нельзя получить чужой ресурс (проверка ownership)
- [ ] **Входные данные ограничены по размеру** — нет неограниченного принятия данных

## 6. Качество кода

- [ ] **`cargo clippy -- -D warnings`** — чистый, без предупреждений
- [ ] **`cargo fmt`** — отформатирован
- [ ] **Тесты есть** — для новой бизнес-логики unit тесты, для endpoint'ов integration тесты
- [ ] **Нет dead code** — `#[allow(dead_code)]` без причины
- [ ] **Документация** — pub API задокументированы `///` комментариями

## Вывод

Выведи результат в формате:

```
🔴 BLOCKER (нельзя мержить):
  - ...

🟡 WARNING (желательно исправить):
  - ...

🟢 INFO (замечания):
  - ...

✅ Итог: [APPROVED / CHANGES REQUESTED]
```
