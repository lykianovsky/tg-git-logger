# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## Проект: GitEye Bot

# 1. Общая информация о проекте

## 1.1 Назначение системы

Universal Dev Activity Bot — это универсальная интеграционная платформа (бот + backend-сервис), предназначенная для
автоматизации процессов разработки.

Система должна:

- Получать события из систем контроля версий (GitHub, GitLab и др.)
- Синхронизироваться с таск-трекерами (Jira, Linear, YouTrack и др.)
- Отправлять уведомления в мессенджеры (Telegram, Slack, Discord, Teams и др.)
- Автоматически управлять статусами задач
- Тегать ответственных пользователей
- Вести журнал всей активности
- Поддерживать заменяемые интеграции без изменения ядра

---

# 2. Цели проекта

- Упростить коммуникацию между разработкой и тестированием
- Исключить ручное переключение статусов задач
- Автоматизировать уведомления
- Повысить прозрачность процессов
- Создать масштабируемую платформу (SaaS / self-hosted)

# 3. Архитектурная концепция

## 3.1 Общий принцип

Система строится по модели:

> Core (Ядро) + Adapter (Интеграции)

## 3.2 Ядро системы (Core)

Ядро отвечает за:

- Обработку входящих событий
- Управление workflow-логикой
- Хранение данных
- Управление пользователями
- Управление ролями
- Управление правилами автоматизации
- Отправку уведомлений через выбранные адаптеры

## 3.3 Интеграционные адаптеры (Adapters)

### Поддерживаемые типы адаптеров:

#### Системы контроля версий:

- GitHub
- GitLab
- Bitbucket
- (расширяемо)

#### Таск-трекеры:

- Jira
- Linear
- YouTrack
- ClickUp
- (расширяемо)

#### Мессенджеры / соц. платформы:

# 5. Интеграция с системами контроля версий

## 5.1 Обработка Webhook

Система должна:

- Принимать webhook-запросы
- Валидировать подпись
- Поддерживать события:

    - push
    - pull_request_opened
    - pull_request_closed
    - pull_request_merged
    - issue_created
    - branch_created
    - release_published

- Логировать:
    - Тип события
    - Репозиторий
    - Проект
    - Пользователя
    - Payload
    - Дату и время

## 5.2 Логирование

- Сохранять всю историю событий
- Связанные задачи и пользователей
- Историю изменения статусов

---

# 6. Интеграция с таск-трекерами

## 6.1 Поиск задачи

Система должна извлекать ключ задачи из:

- Названия PR
- Названия branch
- Коммита

Пример: `PROJ-123 Add login validation`

## 6.2 Автоматическое изменение статуса

Если событие: `pull_request_merged`

То:

- Найти задачу
- Перевести в статус "Testing"
- Обновить поле исполнителя при необходимости

Статус настраивается на уровне проекта.

## 6.3 Тегирование тестировщиков

- Хранить роли пользователей (developer / tester / manager)
- Хранить привязку к проектам
- При переводе задачи в Testing:
    - Упомянуть всех тестировщиков проекта
    - Отправить уведомление

# 12. Основной сценарий работы

1. Пользователь подключает GitHub, Jira и Telegram.
2. Настраивает правило: Pull Request Merged → перевести задачу в Testing.
3. Разработчик мерджит PR.
4. Система:
    - Получает webhook
    - Извлекает ключ задачи
    - Меняет статус
    - Определяет тестировщиков
    - Отправляет уведомление

# 14. Weekly Summary / Activity Digest

## 14.1 Назначение

- Автоматическая сводка активности пользователя или команды
- Подготовка к созвонам
- Прозрачность работы
- Отчёт в 1 клик

## 14.2 Периоды отчётов

- За сегодня, вчера, неделю, месяц, кастомный диапазон

## 14.3 Персональный Summary

- Открытые, смёрженные, закрытые PR
- Коммиты (количество, изменения)
- Завершённые задачи, задачи в работе, задачи в Testing
- Время в статусах

## 14.4 Командный Summary

- Смёрженные PR команды
- Задачи в Testing
- Застрявшие задачи
- Самые активные / менее активные участники
- Среднее время до мерджа

## 14.5 Авто-отправка

- По расписанию
- В личные сообщения, общий чат, email

## 14.6 Пример отчёта в Telegram


- Топ 5 самых больших PR
- Самые долгие ревью
- Просроченные задачи

- Анализ нагрузки по ролям
